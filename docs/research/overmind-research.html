01

연구 배경 및 동기

1.1 문제 정의

Claude Code는 개별 개발자에게 강력한 AI 코딩 보조 환경을 제공하지만, 복수 개발자가 동일 프로젝트에서 협업할 때 근본적인 한계가 존재한다. 각 개발자의 Claude 인스턴스는 독립된 메모리 공간(~/.claude/projects/*/memory/)에서 동작하며, 다른 개발자의 작업 맥락, 트러블슈팅 이력, 설계 결정 과정을 인지하지 못한다.

지식 단절 — 개발자 A가 3시간 삽질 끝에 발견한 해결법을 개발자 B의 Claude는 모르고 동일한 삽질을 반복한다.

Intent 다형성 — PR/merge 후 코드는 하나가 되지만, 각 개발자의 Claude가 기억하는 "왜 이렇게 했는지"가 다르다.

프로젝트 메모리 일관성 붕괴 — 개인적 일관성은 유지되지만, 팀 전체로서의 프로젝트 맥락이 파편화된다.

Claude Code 속도에 의한 괴리 증폭 — Claude Code의 작업량과 속도는 인간 단독 작업 대비 방대하다. 별도 브랜치 작업이 빠르게 진행되어 장기화되거나 merge가 지연되면, 다른 개발자와의 코드베이스 괴리가 기존 Git 워크플로우보다 훨씬 빠르게 확대되며 맥락 불일치가 기하급수적으로 커진다.

1.2 Overmind 비유

StarCraft의 저그 종족에서 Overmind(초월체)는 모든 저그 개체의 의식을 하이브 마인드(Hive Mind)로 통합하는 최상위 존재다. 각 개체는 물리적으로 분리되어 있지만, 하나의 의식체가 전체를 관통한다.

본 연구에서 제안하는 Overmind 아키텍처는 이 개념을 차용한다. 각 개발자의 Claude 인스턴스는 물리적으로 별개의 개체이지만, 지속적 메모리 동기화를 통해 모든 Claude가 동일한 memory pool을 참조하는 상태를 유지함으로써, 어떤 유저가 질문하든 해당 Claude가 전체 팀의 과정과 맥락을 포함한 상태에서 사고하고 답변할 수 있도록 한다.

1.3 핵심 원칙

01
각 작업 주체는 물리적으로 별개 개체다 — 인간 개발자든 자율 에이전트든, 이것은 바뀌지 않는다.
02
자기 memory + 다른 주체의 memory × N을 항상 유지한다 — Overmind의 본질.
03
작업의 지시/자율 판단은 각자가 하되, 사고의 기반은 공유된다 — 지시는 로컬, 맥락은 글로벌.
04
결과뿐 아니라 과정도 공유한다 — 시도, 실패, 이유, correction 전체.
05
다형성을 인지하고 고지한다 — 인간 유저에게 또는 에이전트 자체 판단에 반영.

1.4 적용 범위: 인간 개발자에 국한되지 않음

본 연구는 "복수 개발자"를 주요 시나리오로 서술하지만, Overmind의 적용 범위는 인간 개발자에 한정되지 않는다. 복수의 자율 에이전트가 독립적으로 작업하는 모든 환경에서 동일한 가치가 발생한다.

Claude Code 에이전트 팀, git worktree 병렬 세션, CI/CD 파이프라인의 자율 Claude 인스턴스 등 — 복수 작업 주체가 각자 브랜치에서 독립 작업 후 결론을 통합하는 시나리오에서, 통합 이전에 수시로 과정과 lesson을 공유/전파하면 중복 삽질 방지, 다형성 사전 감지, 암묵지의 사전 방어막 효과가 동일하게 발생한다.

이 문서에서 "개발자"는 "독립적 작업 주체(인간 또는 에이전트)"로 읽을 수 있으며, Overmind의 핵심 가치 — 과정 흡수, 사전 차단, 다형성 감지 — 는 작업 주체의 종류와 무관하게 적용된다.

02

기존 솔루션 분석

Supermemory 활성

SaaS 기반 MCP 서버. containerTag로 메모리 scope 관리. Ontology-aware edge 기반 벡터 그래프 엔진, hybrid search(<300ms), temporal change 처리, contradiction 해소.

Team Memory 기능 있음. Personal/team container 분리. 같은 containerTag 공유 시 팀원 간 메모리 자동 merge.

한계: SaaS 의존(vendor lock-in), pull 기반(실시간성 부족), 과정이 아닌 결론만 공유.

Julep Memory Store archived

2025년 12월 archived. Star 6, fork 0. 실질적 채택 없이 중단. 검토 대상에서 제외.

Claude-Flow / ruflo 활성

Queen Agent 기반 multi-agent swarm. SQLite collective memory.

한계: 한 사람이 여러 agent를 돌리는 구조. 복수 인간 개발자 간 메모리 공유 아님.

Claude Cognitive 활성

Attention 기반 파일 인젝션 + pool 디렉토리를 통한 multi-instance 상태 공유.

한계: 같은 머신 내 여러 세션 간 조율이 주 목적. 복수 머신 시나리오 미지원.

Claude Sync 류 활성

Git repo/encrypted cloud를 통한 ~/.claude/ push/pull 동기화.

한계: 같은 사람의 여러 머신 동기화. 팀 협업 레이어 아님.

핵심 Gap: 기존 솔루션 중 어떤 것도 "복수 독립 작업 주체(인간/에이전트) × 각자의 작업 공간 × 과정 포함 실시간 메모리 동기화 × 다형성 감지"를 동시에 충족하지 못한다.

Simple 접근법과의 비교: MEMORY.md push → project auto-memory 치환

가장 단순한 대안 — 각 개발자의 MEMORY.md를 git push하고, session start 시 skill이 pull하여 project auto-memory로 치환하는 방식 — 과의 정직한 비교가 필요하다.

Simple이 달성하는 것 (70~80%)

auto-memory는 이미 우수한 지식 압축기. git 공유만으로 "A가 뭘 알게 됐는지" 전달 가능. 소규모 팀에서 모듈 겹침 적으면 대부분 해결.

Simple에서 손실되는 것

과정 (결론만 압축), 실시간성 (session end에만 push), 다형성 감지 (의미 충돌 무시), attribution (누구의 맥락인지 소실).

Simple은 Overmind의 "충분히 좋은 v0"다. 실제 시작점은 MEMORY.md push/pull이 맞다. Overmind의 가치는 이 위에 과정 기록, 다형성 감지, 실시간 signal을 점진적으로 쌓아올릴 수 있는 확장 경로가 설계되어 있다는 데 있다.

03

Overmind 아키텍처 설계

각 작업 주체(인간의 로컬 Claude이든 자율 에이전트이든)가 서로 다른 개체이지만, local memory + others memory × N을 계속적으로 sync함으로써, 마치 단일 Claude가 사고를 종합해 답변/분석/과정/결론을 도출할 수 있도록 한다.

3.1 설계 철학

이것은 아키텍처의 변경이 아니라 sync의 빈도와 범위에 대한 운영 조건이다. 각 Claude는 물리적으로 별개이되, 동일한 memory pool을 참조하는 상태를 유지한다.

3.2 핵심 차별점: 결론 흡수가 아닌 과정 흡수

Overmind가 기존 접근법(Simple 포함)과 결정적으로 다른 지점은, 백그라운드에서 결정된 결론을 흡수하는 것이 아니라 의사결정 과정 전체를 흡수하는 것이 목표라는 점이다.

기존 접근법은 "A가 OAuth2를 선택했다"라는 결론만 전달한다. Overmind는 "JWT refresh token rotation 시도 → 복잡도 과다로 기각 → 세션 기반 검토 → stateless 충돌로 기각 → OAuth2 + PKCE 채택"이라는 사고 과정 전체를 B의 Claude가 보유하도록 한다.

B의 Claude가 "캐시 레이어를 auth에 추가하자"는 지시를 받았을 때 — A의 과정을 알고 있는 Claude만이 "A가 stateless 요구사항 때문에 세션 기반을 기각한 전적이 있는데, 캐시 레이어가 사실상 세션 상태를 만드는 것과 같다"고 사전 고지할 수 있다. 결론만 알았다면 이 판단은 불가능하다.

3.3 특수 케이스: Claude Code 속도에 의한 브랜치 괴리

Claude Code의 작업 속도와 볼륨은 인간 단독 개발 대비 방대하다. 이것이 팀 협업에서 새로운 유형의 문제를 만든다.

시나리오

A가 feature-auth 브랜치에서 인증 전체 리팩토링 진행 — Claude 속도 덕에 하루 만에 수십 개 파일 변경 + 테스트 + 마이그레이션 + 문서 생성. 동시에 B는 feature-cache 브랜치에서 성능 최적화 진행. 양쪽 모두 Claude가 빠르게 작업하므로, 코드베이스 괴리가 인간 페이스보다 훨씬 빠르게 확대. merge가 1~2일만 지연되어도 충돌 지점이 기하급수적으로 증가.

더 중요한 것은 코드 충돌보다 맥락 충돌 — A의 Claude는 "인증은 OAuth2 기반"이라는 전제로 사고하고, B의 Claude는 "인증은 여전히 JWT 기반"이라는 전제로 캐시를 설계.

Overmind의 해법: Signal axis를 통해 A의 브랜치 작업이 수시로 B의 Claude에게 전달. B의 Claude는 merge 전에 이미 "A가 인증 체계를 전면 교체하고 있다"를 인지. Claude Code의 속도가 만들어낸 문제를 Claude의 인지 능력으로 해결하는 구조 — 인간 개발자 간의 구두 소통("나 지금 auth 건드리고 있어")을 Claude가 자동으로 대행.

3.4 핵심 해자(Moat): 문제 발생 이전의 사전 차단

Overmind의 가장 결정적인 가치는 문제가 발생하기 전에 사전 차단할 수 있다는 점이다. 이것이 단순한 지식 공유를 넘어서는 핵심 해자다.

대표 시나리오: 환경 설정 문제의 사전 차단

백엔드 서비스 A-B-C 간 설정값 처리가 main에 미처 반영되어 있지 않거나, .env 로컬 설정을 개별 유저가 직접 해야 하는 상황. 코드 리뷰에서도 잡히지 않고, 문서화도 누락되기 쉽고, 각 개발자가 독립적으로 부딪히고 독립적으로 해결하는 패턴이 반복된다.

기존 워크플로우

B가 연결 오류 발생 → Claude Code로 디버깅 → 30분~1시간 후 원인 발견(.env 설정 누락) → memory에 기록, push → 다음 날 A가 동일 문제를 마주침 → 같은 삽질 반복

Overmind 워크플로우

B가 해결, 과정 포함 push → A가 해당 문제를 마주치기 이전에 관련 작업 프롬프팅 시 sync → A의 Claude가 서비스 B 작업 진입 순간 에러 발생 전에 사전 고지 → A는 에러를 한 번도 보지 않고 설정 완료

핵심 차이는 타이밍이다. 기존: "문제 발생 → 검색 → 해결". Overmind: "sync된 memory → 관련 작업 진입 시 사전 체크 → 문제 발생 자체를 방지".

이 패턴은 환경 설정에만 국한되지 않는다. 특정 API의 숨겨진 제약사항, 라이브러리 버전 호환 이슈, 배포 환경 주의점 등 — 한 명이 삽질로 발견한 모든 암묵지가 팀 전체의 사전 방어막이 된다.

"결론 공유"로는 불가능하고 "과정 공유"여야만 가능한 이유: 결론(".env에 이 값을 넣어라")만 있으면 Claude가 사전 고지의 적절한 타이밍을 판단할 맥락이 부족하다. 과정("서비스 B 실행 시 연결 오류가 났고, 원인이 이 설정 누락이었다")까지 있으면 Claude가 "지금 유저가 서비스 B를 다루려 하고 있으니, 이 시점에 알려야 한다"고 판단할 수 있다.

3.5 적용 범위: 인간 개발자에서 자율 에이전트까지

본 문서의 시나리오는 "인간 개발자 A, B"를 기준으로 서술하나, 이는 설명 편의이며 아키텍처 제약이 아니다.

복수 자율 에이전트 시나리오

Agent-A(백엔드 API) / Agent-B(프론트엔드) / Agent-C(테스트)가 독립 수행 시에도: 과정 공유(API 스키마 3번 변경 이유), 사전 차단(인증 요구사항 선전파), 다형성 감지(같은 모델에 다른 전제)가 동일하게 적용.

측면인간 개발자자율 에이전트
지시유저 프롬프트오케스트레이터 / 자체 판단
고지 대상인간에게 자연어에이전트 context에 injection
Push/Pull 빈도SessionEnd 위주Task 완료마다 (더 빈번)

에이전트 시나리오에서는 push/pull 빈도가 높아지므로 서머리/lesson 추출이 더 중요해진다. context window 효율성에 민감한 에이전트에게 raw JSONL보다 lesson 수준 압축이 기본.

3.6 2-Axis 설계

Axis 1: Signal (실시간)

"지금 뭘 하고 있는지"
JSONL type: change, correction(진행중). 실시간, 휘발적.

Axis 2: Knowledge (누적)

"뭘 배웠는지"
JSONL type: decision, discovery, correction(완료). 누적적. 두 축 모두 동일 JSONL로 저장, 클라이언트가 해석 시 구분.

3.7 Relay의 실체: Overmind Server

Overmind server는 로컬 또는 사내에서 직접 구동하는 HTTP(MCP) 서버다. 외부 SaaS가 아닌, 팀의 작업 집약을 위한 자체 운영 서버.

Push (클라이언트 → 서버)

Push 내용은 MEMORY.md가 아닌 JSONL 원시 이벤트 로그. 프롬프트, 과정, 결과를 구조화된 로그로 전송. 트리거: SessionEnd, Correction 감지, 주요 구조 변경.

Pull (서버 → 클라이언트)

서버가 서머리/핵심(lesson)을 추출하여 전달 (경량 LLM). ?detail=lesson|summary|full로 상세도 선택. 고도 판단(다형성, 사전 차단)은 클라이언트 Claude의 몫.

claude mcp add overmind http://내부서버/mcp로 즉시 연결. MCP 생태계 자연 통합. 사내 네트워크 한정 접근으로 보안 자연 확보. Git repo 오염 없이 서버가 memory의 single source of truth.

서버 측 Housekeeping: JSONL 보관, scope 인덱싱, TTL 만료, 중복 제거. 그리고 pull 시 서머리/핵심 추출 — 경량 LLM이면 충분하므로 서버에 위임하기 적합. 고도 판단(다형성 감지, 사전 차단)은 클라이언트 Claude의 몫.

Push는 전부, Pull은 핵심만. 클라이언트는 raw JSONL을 가공 없이 전부 push. 다른 클라이언트가 pull할 때 서버가 서머리/lesson만 추출하여 전달. 초기에는 lesson(확정 결론)만 → 점차 summary(과정 포함) → full(raw 접근)으로 확대.

3.8 Memory 병합과 Conflict-free 설계

Conflict는 구조적으로 발생할 수 없다. Memory는 코드가 아니므로 conflict 개념 자체가 적용되지 않는다.

01
각 유저의 memory는 개별 파일(memory-sync/user_a/, user_b/)로 격리 — 파일 레벨 conflict 불가능
02
project-memory/로의 통합은 Claude의 memory 정리 스킬이 수행 — append + deduplicate + summarize이지 two-way merge가 아님
03
모순되는 내용("A는 nginx 안 됨, B는 nginx로 해결")은 conflict가 아닌 다형성으로 처리 — 환경/조건에 따라 다를 수 있는 지식으로 병합

3.9 Scope: Repo-scoped

Overmind는 개별 사용자의 Claude 전역 내용을 통합하는 것이 아니다. 특정 프로젝트 repo 단위의 연결을 지향한다.

3.12 명시적 공유 Push (Explicit Broadcast)

Hook 자동 push 외에 의도적으로 lesson/변경사항을 공유하는 명시적 push가 필요하다.

필요한 시나리오

01
마스터 에이전트 설계 변경 — API 스키마 변경을 이미 구현 중인 worker들에게 즉시 전파
02
구현 중 긴급 발견 — 프론트 에이전트가 발견한 인증 제약을 백엔드 에이전트에게 즉시 공유
03
인간의 의도적 공유 — "이 삽질 결과를 팀 전체가 알아야 해"라는 명시적 판단

Hook 자동 push

Hook 이벤트로 트리거. 자동 캡처된 로그. 일반 우선순위. 다음 pull 시 수신.

명시적 broadcast

유저/에이전트의 의도적 지시. priority: urgent 가능. 다음 pull 시 urgent 우선 수신. MCP tool로 "팀에 알려줘"로 트리거.

두 종류 모두 동일 JSONL 포맷으로 저장. 차이는 type과 priority 필드뿐. 클라이언트가 priority 기반 정렬로 urgent를 먼저 처리.

구현: Claude Code 플러그인 조합으로 완전히 가능

컴포넌트역할
.mcp.jsonOvermind 서버의 push/pull/broadcast를 Claude tool로 노출. 실제 전송 채널.
hooks.jsonSessionStart→자동 pull, SessionEnd→자동 push, PreToolUse→관련 영역 pull. systemMessage로 urgent를 context 직접 injection.
Skill (SKILL.md)"팀에 알려줘" 같은 자연어에 자동 매칭 → MCP broadcast tool 호출.
Slash Command/overmind:broadcast로 명시적 트리거.

비동기 실시간 수신(SSE/WebSocket)은 불필요하다. Claude Code가 세션 간 실시간 push를 네이티브로 지원하지 않으므로, Hook 트리거 시점(SessionStart, PreToolUse)에서의 pull이 수신의 자연스러운 타이밍. "다음 Hook 발생 시 urgent 우선 수신"이면 실용적으로 충분하며, SSE 레이어 추가의 복잡성 대비 이점이 없다.

3.15 다형성(Polymorphism) 감지

트리거 조건

같은 파일 경로 또는 같은 모듈에 대해 서로 다른 user의 signal이 존재하는데, intent가 의미적으로 다를 때. 받는 쪽 Claude가 자연어 비교 판단으로 수행 — 벡터 유사도 같은 무거운 인프라 불필요.

Claude의 역할: 판단이 아니라 고지

"auth 모듈에 대해 A 개발자는 보안 강화 관점(JWT → OAuth2 전환)으로 작업했고, 현재 세션에서는 성능 최적화 관점(세션 캐시 레이어 추가)으로 접근하고 있습니다. A가 stateless 요구사항 때문에 세션 기반을 기각한 전적이 있는데, 캐시 레이어가 사실상 세션 상태를 만드는 것과 유사합니다. 이 부분을 먼저 확인하는 게 좋겠습니다."

이 수준의 고지는 결론만 알았다면 불가능하다. A의 과정 전체를 알고 있기 때문에 가능한 판단이다.

marketplace 없이 로컬 로딩. /reload-plugins로 hot-reload. Agent SDK로 headless 테스트도 가능.

5.2 Push/Pull 데이터의 가치 자기평가

의도적 lesson 전파가 잘 이루어지고 피드백 점수가 확보되는지가 Overmind의 핵심 성공 기준이다. "전파"란 서버에 도착이 아니라, 다른 클라이언트의 Claude가 실제로 인지하고 반영하는 것까지. 피드백이 기록되어야 선순환 구조가 작동한다.

동기적 관련성 체크

Pull 직후 PostToolUse Hook에서 경량 모델로 관련성 1~5점 평가. 결과를 서버로 재push → 이벤트별 유용성 통계 축적.

회고적 품질 평가

Session End에서 주입된 memory 중 실제 참조/무시 분류. prevented_error: true로 사전 차단 정량 측정.

선순환 구조: 서머리 품질 ↑ → relevance_score ↑ → 더 자주 전달 → 더 많은 사전 차단 → 피드백 반영 → 서머리 품질 ↑

해결해야 할 과제

Relay의 분석 역할 — 해결됨

Relay가 로컬 Claude의 memory 정리 스킬과 동일한 방식으로 housekeeping 수행. 중복 제거, 요약, 유효성 판단을 relay 단에서 처리. (§3.6 참조)

Conflict Resolution — 해결됨 (구조적 불가)

Memory는 코드가 아니므로 conflict 개념이 적용되지 않음. 각 유저 memory는 개별 파일로 격리, project-memory 통합은 Claude의 memory 정리 스킬이 수행. 모순은 다형성으로 처리. (§3.7 참조)

Noise vs. Signal — 부분 해결

클라이언트 위임 설계로 noise 판별 주체가 pull하는 클라이언트 Claude가 됨. 잔여: push 시점의 최소 필터링(Hook 단에서 correction/decision/discovery만 이벤트화). PoC에서 조정.

Context Window 압박 — 부분 해결

Scope 기반 pull + 클라이언트 측 관련성 판단으로 불필요한 이벤트 무시. 잔여: scope 내 이벤트 과다 시 TTL + 최근 N개 제한 전략. PoC에서 실측 후 조정.

보안 및 프라이버시 — 부분 해결

Overmind server 방식으로 git repo 오염 해소. 사내 네트워크 한정 접근으로 외부 노출 차단. 단, push 시 민감 정보 필터링은 클라이언트 측에서 필요.

07

결론

본 연구는 Claude Code 환경에서 복수의 독립적 작업 주체(인간 개발자 또는 자율 에이전트) 간 메모리 동기화라는 미개척 영역에 대해, 기존 솔루션의 한계를 분석하고 Overmind라는 실현 가능한 아키텍처를 제안하였다.

Overmind의 본질은 아키텍처의 복잡성이 아니라 sync의 빈도와 범위에 있다. 각 Claude는 물리적으로 별개 개체이되, 자기 memory + 팀원 memory × N을 지속적으로 동기화함으로써, 어떤 유저가 질문하든 해당 Claude가 전체 팀의 과정과 맥락을 포함한 상태에서 사고하고 답변한다.

차별점

01
과정 흡수 — 백그라운드에서 결론만 흡수하는 게 아닌, 시도→실패→correction→선택의 전체 과정. 결론만 알면 "뭘 해야 하는지"는 알지만 "뭘 하면 안 되는지"는 모른다. 과정을 알면 둘 다 안다.
02
Claude Code 속도 환경 대응 — Claude Code의 방대한 작업량이 만드는 브랜치 괴리 증폭을, 작업 중 실시간 맥락 공유로 해결. 인간 간 구두 소통을 Claude가 자동 대행.
03
다형성 인지 — intent 충돌을 감지, 유저에게 고지, 함께 reconcile.
04
Conflict-free + 지능 분산 — memory는 코드가 아니므로 conflict 개념이 적용되지 않는다. 모순은 다형성으로 처리, relay가 Claude의 memory 정리 스킬로 housekeeping 수행.
05
Self-hosted / Repo-scoped — SaaS 의존 없이 프로젝트 단위 동작.
06
점진적 확장 경로 — Simple(MEMORY.md push/pull)에서 시작 → 과정 기록 → 다형성 감지 → 실시간 signal로 점진적 확장 가능한 설계.
06
인간/에이전트 불문 — 인간 간 협업뿐 아니라 복수 자율 에이전트의 병렬 작업 + 결론 통합에서도 동일 작동하는 범용 설계.

핵심 해자: 사전 차단

Overmind의 가장 결정적인 해자는 문제 발생 이전의 사전 차단이다. 한 개발자가 삽질로 발견한 암묵지가 과정 포함으로 공유되면, 다른 개발자의 Claude는 관련 작업 진입 시점에 에러 발생 전에 선제 고지할 수 있다. 기존 "문제 발생 → 검색 → 해결"이 아닌, "sync된 memory → 작업 진입 시 사전 체크 → 문제 발생 자체를 방지". 이것은 과정까지 공유되어야만 Claude가 고지의 적절한 타이밍을 판단할 수 있다.

B

부록: 장기 비전 — Overmind의 이름값

현재 Overmind server는 JSONL 중계소에 가깝다. 저장은 일시적이고, 지능은 edge에 집중되어 있으며, 서버 자체는 "사고"하지 않는다. Overmind라는 이름에 걸맞으려면 최종적으로 서버가 중앙 집중적 분석 능력을 가져야 한다.

장기 과제: 중앙 분석 레이어

01
Repo-level analysis — 프로젝트 repo를 직접 분석하여 JSONL과 실제 코드 변경을 대조. 아키텍처 패턴 추출, 기술 부채 추적.
02
Cross-user insight — 전체 유저 이벤트를 종합하여 "이 영역에 혼란이 집중", "3명이 서로 다른 전제로 작업 중" 같은 팀 레벨 인사이트.
03
Proactive guidance — 서버가 "이 유저는 지금 이 정보를 알아야 한다"고 판단하여 능동적 push. 진정한 하이브 마인드의 중추.

클라이언트 위임 모델은 유지하되, 서버 측 분석 레이어를 점진적으로 추가하는 방향. JSONL이 이미 구조화되어 흐르고 있으므로 자연스러운 진화.
